# Grove Game Code Audit — Dead Code & Design

_Audited 44 runtime source files. 86 confirmed dead-code items, 3 high- and 40 medium-severity design issues. tools/ excluded._

## Executive summary

The grove codebase is structurally sound at the leaf level but rots at two load-bearing trunks. `engine/scripts/scenes/board.gd` (2175 lines, ~90 methods) and `engine/scripts/scenes/map.gd` (1738 lines) are god-objects that fuse layout, raw input + a hand-threaded drag state machine, merge/sell/stash actions, generator-burst economy, quest fences, monetization offers, FTUE spotlights, and save plumbing into ~50 loose mutable members each. There is no internal module boundary — any method can poke any field. This is the single biggest source of risk; everything else is downstream.

The engine/grove boundary is a fiction. `engine/scripts/core/save.gd` is billed "theme-agnostic persistence" but hardcodes grove's entire schema plus a literal grove spot-id rename table (`_SPOT_ID_RENAMES`: `fh_chest`→`fh_hearth`, etc.). `content.gd` ("content ENGINE, theme-agnostic") hand-mirrors ~70 grove DATA consts that must be edited in lockstep, and `tuning.gd` bakes in grove hex colors and art filenames. `games/active.gd`'s own header concedes the engine is compiled against one game's ruleset and the roster has exactly one entry. The generic-engine pretense costs maintenance and misleads readers.

`save.gd` is also carrying a fully dead persistence schema from a previous game — `jobs/rooms/clients/stats` dictionaries seeded into every save, with a full live API (`record_job`, `room_decor`, `buy_decor`, `boards_cleared`, plus `bump_daily`/`claim_daily`) that no runtime code calls. `save_tests.gd` (referencing "bedroom_01", "rug") keeps it looking alive. That's ~120 dead lines written into every real grove save file. Separately, every mutator ends in `save_now()`, which stringifies + writes + re-parses the entire save synchronously; `content.earn_stars` does up to four full-file writes for one star award.

Two correctness bugs surfaced, both worth fixing regardless of refactors: `merchant_stand.gd` advertises the wrong tier on its sell pill (promises a premium gem, shows banded coins), and `ads.gd` writes `Save.ad_record` before validating the ad type, so an in-DATA-but-unhandled type burns a daily cap slot with no reward.

What's genuinely well done: the pure-logic layer (`board_model.gd`, `board_logic.gd`, `quests.gd`, `content.gd` formulas) is properly separated and testable; `grove_data.gd` is cleanly data-only; the save write path verifies durability by re-parsing; `design.gd` and `level_popup.gd` came back clean. The deterministic intake-script split and the per-suite test runner are real assets.

Overall health: medium. No crashes-waiting-to-happen at scale, but two unmaintainable god-objects and a leaky boundary make every grove change costlier than it should be. 86 confirmed-dead symbols and 40 medium issues are mostly mechanical.

If you fix five things, fix these:
1. Delete the dead `jobs/rooms/clients/stats` schema + accessors + tests in `save.gd` (~120 lines, zero risk).
2. Fix the two real bugs: `merchant_stand.gd` sell-pill tier and `ads.gd` validate-before-record ordering.
3. Extract a `DragController` and a `BoardView` grid-builder out of `board.gd` to establish the first real internal seam.
4. Batch `save.gd` writes behind a dirty-flag/coalesced commit; collapse `earn_stars`' four writes into one.
5. Decide the engine/grove boundary honestly — either move grove persistence (esp. `_SPOT_ID_RENAMES`) into a `games/grove` save-extension, or drop the "theme-agnostic" labels in `save.gd`/`content.gd`/`tuning.gd`.

## Cross-cutting architecture

### [HIGH] board.gd and map.gd are two ~2000-line god-objects mixing layout, input, model, economy, persistence, and FTUE/monetization
*Files:* engine/scripts/scenes/board.gd, engine/scripts/scenes/map.gd

board.gd (2175 lines, ~110 members/funcs) is the single coordinator for: scene layout (_make_board_mat/_make_board_frame/_make_branch_divider/_field_backdrop), the HUD + water-refill stack (_build_hud/_build_water_hud/_update_water_hud), raw pointer input + drag state machine (_on_board_input/_on_press/_on_release/_release_gen with 9 _drag_* / _press_* members), the merge/move/swap/sell/stash actions, generator burst-pop economy (_pop_seed, _gen_burst_level, _upgrade_gen_burst), quest fence generation + giver lights (_refill_quests/_rebuild_givers/_refresh_giver_lights), the merchant basket + porter timer (_porter_tick), spotlight FTUE (_maybe_spotlight_chrome), monetization (_on_ad_refill/_on_oow_offer/_grant_oow_offer), and save plumbing (_load_state/_persist). map.gd (1738 lines) similarly fuses map/select/welcome view-routing, shader-clipped art composition (_HOME_MASK_SHADER, _ART_CLIP_SHADER, _veil_thumb), HUD, the liveops rail (daily/free-gems/inbox/vault badges), and persistence. State is held as ~50 loose scene members each (board.gd lines 81-154), so any method can mutate any field — there is no internal module boundary. The drag state alone (board.gd:137-142, _press_cell/_press_pos/_drag_is_gen/_drag_node/_drag_from/animating) is threaded by hand through five methods.

**Fix:** Extract cohesive collaborators that the scene owns but does not inline. Natural seams in board.gd: (1) a DragController owning _press_*/_drag_* + _on_press/_on_release/_release_gen and emitting intents (merge/move/sell/stash); (2) a BoardView/grid-builder owning _make_slot/_make_board_mat/_make_board_frame/_cell_pos/_pos_to_cell/_rebuild_pieces; (3) a WaterHud/refill-wall unit owning _build_water_hud/_update_water_hud/_on_refill/_on_ad_refill/_on_oow_offer; (4) a QuestFence unit owning _rebuild_givers/_refresh_giver_lights/_refill_quests. In map.gd: split view-routing (_open_map/_open_select/_build_select) from map-art composition (the shaders + _build_map_base/_seat_spots/_veil_thumb) and from the liveops rail. The static board_model/board_logic/quests already prove the pattern — push more pure logic down to them and keep the scene as a thin wiring layer.

### [HIGH] The 'generic engine' is saturated with grove specifics — the engine/grove boundary is not clean
*Files:* engine/scripts/core/save.gd, engine/scripts/core/content.gd, engine/scripts/core/tuning.gd, games/active.gd

The split is asymmetric: grove_data.gd is cleanly data-only, but the supposedly theme-agnostic engine hard-codes grove concepts. save.gd — billed as 'theme-agnostic persistence' (line 2) — embeds the grove's entire schema: a grove() blob accessor, residents keyed by grove map ids, bag-slot/burst/spotlight/ad-ledger/oow/vault/water-pending accessors, and crucially _SPOT_ID_RENAMES (save.gd:268-273) which hardcodes literal grove spot ids ('fh_chest'->'fh_hearth', 'bn_doors'->'bn_bales', etc.) and _MAP_KEY_RENAMES. A second game could not reuse save.gd without inheriting grove's migration table. content.gd is named the 'content ENGINE (theme-agnostic)' (line 2) yet re-exports ~70 named grove DATA consts (content.gd:15-80) and its comments reference grove §-sections throughout. tuning.gd class Shop/Hud bake in grove palette hex values and grove art filenames ('quest/card_quest.png', grove honey/leaf colors). active.gd's own header (lines 4-9) concedes the leak: DATA+PALETTE 'can't be a per-run env layer' so the engine is compiled against ONE game's ruleset; only art/audio/font actually layer at runtime. The 'roster' has exactly one entry (active.gd:17-19).

**Fix:** Either (a) drop the generic-engine pretense and rename/co-locate these as grove modules (honest, low-risk, removes misleading comments), or (b) if multi-game is a real goal, move the grove-specific persistence (grove blob accessors + _SPOT_ID_RENAMES + residents/bag/burst/vault) out of engine/core/save.gd into a games/grove save-extension that composes a small generic Save core (currencies + atomic file IO + versioning only). The spot-id rename table in particular must not live in the engine — it is pure grove content migration.

### [MEDIUM] save.gd carries a dead persistence schema from a previous game ('Tidy Up rooms') alongside the live grove blob
*Files:* engine/scripts/core/save.gd

_default() (save.gd:31-41) seeds every save file with jobs/{}, rooms/{}, clients/{}, stats/{boards_cleared} plus NEW_SAVE_GEMS — and save.gd exposes a full API around them: record_job/job/clear_paid (lines 145-166), room_decor/decor_owned/decor_count/buy_decor (172-194), boards_cleared/record_board_clear (136-143), and the legacy progress.cfg migration that seeds coins from past 'board clears' (60-67, COINS_PER_CLEAR_SEED). Grep confirms NONE of these are called by any runtime code (engine/scripts or games, excluding tools/tests) — they are exercised only by engine/tests/save_tests.gd (which references 'bedroom_01'/'bedroom'/'rug' decor — a different game's vocabulary). The live grove game stores everything in the separate grove() blob (board/bag/unlocks/quests/residents/...). So every grove save file writes and round-trips four top-level dead dictionaries plus a dead BAG_MIN/MAX const duplicated from grove_data, and a passing test suite keeps the dead code looking alive.

**Fix:** Delete the jobs/rooms/clients/stats schema, their accessors (record_job/job/clear_paid/room_decor/decor_owned/decor_count/buy_decor/boards_cleared/record_board_clear), the progress.cfg legacy migration (or confirm whether any shipped build actually has progress.cfg files in the wild first), and the corresponding save_tests.gd cases. This removes ~120 lines of save.gd and clarifies that the grove blob is the only real schema. If progress.cfg migration must survive for shipped installs, isolate it and comment why.

### [MEDIUM] Save writes are unbatched: every currency/state mutation re-serializes and re-writes the whole save file to disk
*Files:* engine/scripts/core/save.gd, engine/scripts/scenes/board.gd, engine/scripts/core/content.gd

Each mutator (add_coins/spend/add_diamonds/add_stars/set_bag_slots/ad_record/...) ends with save_now() (save.gd:84-99), which JSON.stringify's the ENTIRE data dict, writes a temp file, re-parses it to verify, then does two DirAccess renames — synchronously, per call. content.earn_stars (content.gd:586-600) calls Save.add_stars (1 write) then add_diamonds (another write) then Vault.skim (another) then grove_write (another) — up to 4 full-file writes for one star award. board.gd calls _persist() at 27 sites and Save mutators at 14 more; a single generator burst (_pop_seed) writes the file once via _persist plus more on reward. On a large grove save (board + bag + residents across 5 maps + ad ledgers) this is a lot of redundant disk churn and a re-parse-verify on every coin tick. The verify-on-write is good for durability but pays the full cost every mutation.

**Fix:** Add a dirty-flag / coalesced-write path: mutators mark data dirty and an explicit commit() (or a short debounce / end-of-frame flush) does the single atomic write. Keep an immediate-flush option for true checkpoints (purchase, level-up). At minimum, batch the multi-write sequences like earn_stars behind one write. This also removes the temptation to sprinkle _persist() defensively.

### [MEDIUM] Cross-scene and cross-node coupling via static vars and get_parent() chains rather than explicit interfaces
*Files:* engine/scripts/scenes/board.gd, engine/scripts/scenes/map.gd, engine/scripts/ui/hud.gd

board.gd hands a value to map.gd by writing a static var across the scene transition: HomeScene.decorate_map = _decorate_target() then change_scene (board.gd:246-247), consumed and cleared in map._ready (map.gd:42, 182-186). This is a hidden global channel between two 2000-line files, only documented by a comment and a test. Separately, the HUD contract is a positional reach-through: hud.gd's own comment (line 52) states 'scenes resolve the wallet panel as stars_label.get_parent().get_parent()', and both scenes + the test suites depend on that exact two-level ancestry (grove_ui_tests.gd:79-80, grove_placement_tests.gd:48-63). Any change to the HUD's internal node nesting silently breaks callers that walk up two parents from a label. Drop targets are likewise resolved by global-rect hit-testing against sibling buttons (board.gd:1388-1393, bag_btn/merchant_btn.get_global_rect().has_point) rather than a registered drop-target list.

**Fix:** Replace the decorate_map static with an explicit scene-transition payload (e.g. a small autoload/transition param object, or pass via a Game navigation helper). Have Hud.build return the wallet panel ref directly (it already returns a dict — add hud.wallet, which board.gd already reads as hud.wallet at board.gd:461) and forbid get_parent().get_parent() walks in both scenes and tests. Register drop targets explicitly instead of hit-testing named sibling members.

### [LOW] content.gd re-exports ~70 DATA constants by hand — a brittle, duplicated indirection layer
*Files:* engine/scripts/core/content.gd, games/grove/grove_data.gd

content.gd lines 15-80 manually mirror every grove_data const (COLS=D.COLS, ROWS=D.ROWS, ... SPOTLIGHTS=D.SPOTLIGHTS) purely so existing callers can keep writing G.X and := inference resolves. This is ~65 lines of pass-through that must be edited in lockstep with grove_data: add a tuning knob to grove_data and you must also add its re-export line here or G.NEW is undefined. MAPS is special-cased as a static var (content.gd:70) because grove_data builds it lazily, while the rest are consts — an inconsistency that readers must know. The comment admits the reason is a GDScript limitation (consts can't be reached through a runtime var), so the indirection is structural, but the hand-maintained mirror is the fragile part.

**Fix:** Consider having callers read G.D.X (one const D = Game.DATA already exists at content.gd:14) instead of re-exporting each name, or generate the re-export block, or at least add a test that asserts every grove_data const has a matching content.gd re-export so a missing one fails loudly instead of at runtime. Low priority — it works — but it is a standing maintenance tax and a silent-break risk.

### Duplication hotspots
- _build_hud / _update_hud and the wallet+level-chip wiring are near-identical in both scenes (same Hud.build opts shape, same field captures stars_label/coins_label/level_label/_open_shop, same water_grant closure pattern). The shared Hud module already centralizes the widget, but each scene re-implements the opts assembly and the wallet/level resolution independently, and both depend on the same get_parent().get_parent() ancestry contract. — _engine/scripts/scenes/board.gd, engine/scripts/scenes/map.gd, engine/scripts/ui/hud.gd_
- _load_state / _persist + the grove()-blob read/write dance is duplicated per scene with divergent subsets (board.gd reads board/quests/bag/water/rng + seeds gates; map.gd reads unlocks/last_map). Both poke Save.grove() directly and call Save.grove_write(), and both run ad-hoc pre-§7/pre-T17 migrations inline (board.gd:337-345 gate-backfill, map.gd:222-226 last_map sanitize) rather than in the persistence layer where the other migrations live. — _engine/scripts/scenes/board.gd, engine/scripts/scenes/map.gd, engine/scripts/core/save.gd_
- Progression query wrappers are triplicated: content.gd defines map_spots_done/map_unlocked/owned_count/map_stars_left/frontier_map/is_cheapest_open as static helpers, map.gd re-wraps each one as a thin instance method binding `unlocks`/`_gates()` (map.gd:239-258), and quests.gd defines its own current_map/map_done/meter_target/gate_ready over the same unlocks+gates inputs. Three layers compute the same map-completion predicates. — _engine/scripts/core/content.gd, engine/scripts/scenes/map.gd, engine/scripts/core/quests.gd_
- giver_stand.gd and merchant_stand.gd are sibling stand builders with shared anatomy; merchant_stand already imports GiverStand and reuses ask_pill() + bob(), but each independently re-creates the bust placement, the wicker/treat StyleBoxFlat panels, and the pill row layout. The two could share a common 'fence stand' base for the bust+pill skeleton instead of merchant reaching into GiverStand for two helpers while duplicating the rest. — _engine/scripts/ui/giver_stand.gd, engine/scripts/ui/merchant_stand.gd_
- The daily-usage + cooldown ledger pattern is implemented twice in save.gd with the same {day, used, last} shape and day-rollover logic: _ad_row/ad_used_today/ad_can_show/ad_cooldown_left/ad_record (lines 458-499) and _oow_row/oow_used_today/oow_can_show/oow_record (lines 542-567). The oow block is a hand-copy of the ad block specialized to one key. — _engine/scripts/core/save.gd_

### Worst offenders
- **engine/scripts/scenes/board.gd** — 2175 lines, ~50 loose scene-state members and ~90 methods spanning layout, a hand-threaded drag state machine, merge/sell/stash actions, generator burst economy, quest-fence generation, merchant basket + porter, water-refill wall, monetization offers, FTUE spotlights, and save plumbing — the single point where every grove subsystem is wired with no internal module boundaries.
- **engine/scripts/scenes/map.gd** — 1738 lines fusing view routing (map/select/welcome), shader-based art composition (two inline GLSL shaders + veil/clip/mask), the HUD, the full liveops rail (daily/free-gems/inbox/vault badges + popups), resident welcoming, and persistence — second god-object, and it shares state with board.gd via a static var.
- **engine/scripts/core/save.gd** — Billed as theme-agnostic persistence but hardcodes grove spot-id rename tables and the entire grove schema, AND still carries a fully dead 'rooms/jobs/clients' schema from a previous game; every mutator also does a synchronous full-file stringify+write+reparse, so it is both the boundary-violation hotspot and the I/O-amplification hotspot.

## Design issues by file

### `/Users/xup/dh/merge/engine/scripts/scenes/board.gd`
*The Board scene coordinator (attached to engine/scenes/Board.tscn): owns the persistent merge board model, renders cells/pieces/generators/brambles, drives all input (tap/drag/merge/move/swap/sell/stash), the quest-giver fence, water/energy economy, refill+ad+IAP offers, the bag, the merchant sell+buy-back, and the §14 spotlight. A large god-object that wires together ~25 preloaded modules.*

- **[HIGH · god-object] God-object scene coordinator (~2175 lines, ~80 methods)** (L1-2176)
  - board.gd owns the model lifecycle, every render path, all input/drag state, the quest fence, water economy, three monetization surfaces (refill / watch-ad / IAP offer), the bag, the merchant sell + buy-back + porter, and the §14 spotlight. State is spread across ~50 member vars with cross-cutting invariants (e.g. _drag_node is shared between the board-piece drag path and the bag drag path via _bag_drag_idx). Any change risks touching unrelated subsystems; the file is hard to test in isolation and is where most of the dead code below accumulated because retired features left their members and guards behind.
  - _Fix:_ Extract cohesive collaborators that already have natural seams: a Water/refill controller (water, _regen_ts, refill_btn/ad_refill_btn/oow_offer_btn and their handlers), a Merchant/basket controller (basket, _porter_*, _sell_item/_grant_sale/_record_sale/_buy_back/_rebuild_basket), and a DragController. Each can hold its own state and be unit-tested without instancing the whole scene.
- **[HIGH · fragile-pattern] Retired-feature debris: dead members + permanently-false guards** (L104,108,111,121,130,659,739,755-762,789-796,869-873,2076-2096,2119-2143)
  - Several subsystems were 'retired to a bottom-nav well' but their old node members (merchant_chip, merchant_rest, _amb_layer, basket_chip, bag_slots_ui) were left declared and are now permanently null/empty. The code is littered with `if X != null and is_instance_valid(X)` guards that can never be true, plus whole method bodies (_play_porter, the spirit loop in _buy_treat, the basket-UI build in _rebuild_basket) that never execute. This is a correctness trap: a reader cannot tell which guarded blocks are live, and the buy-back feature LOOKS implemented (basket array fills, sale records, porter timer ticks) but renders nothing because basket_chip is only set by the dead _make_merchant_stand.
  - _Fix:_ Delete the dead members and their guarded blocks, or — if buy-back is meant to ship — wire basket_chip from the live _make_merchant_button path so _rebuild_basket actually renders. Right now the sell-grant works but the advertised buy-back window is unreachable in the game.
- **[MEDIUM · duplication] Generic 'find drop target by global-rect' drag plumbing duplicated across three release paths** (L1368-1450,1806-1820)
  - _on_release, _release_gen, and _end_bag_drag each independently recompute a global point and test bag_btn.get_global_rect().has_point(gp) / merchant_btn.get_global_rect().has_point(gp) with slightly different transforms (board_area.get_global_transform() * pos vs affine_inverse() * gpos). The bag/merchant hit-test logic is copy-pasted between _on_release (1388-1394) and _release_gen (1431-1443) with divergent behavior (gen refuses sell, piece sells). A change to where the bag/merchant wells live, or to the drop-priority order, must be made in multiple places and is easy to get inconsistent.
  - _Fix:_ Factor a single `_drop_target_at(global_point) -> {kind, ...}` helper that both release handlers consult, so the well hit-testing and priority order live in one place.
- **[MEDIUM · encapsulation] Deep reach-through into Save.grove() dictionary with string keys** (L313-345,364-374,385-396,420-432,506,1497,1506,1534,1545,1886,1889)
  - board.gd repeatedly pokes the raw persistence blob: Save.grove().get("unlocks",{}), grove()["seen"], grove()["pops"], grove()["burst_lvl"], grove()["gates"], grove()["water"], etc., and even mutates it in place (g["seen"][str(code)] = true at 370; gg["gates"]=mg + Save.grove_write() at 343-345; g["pops"] = ... at 1506). The schema of the save dict is implicit and duplicated across board.gd, map.gd, and Save. A key typo or schema change is a silent data bug with no single source of truth, and tests must hand-craft the same dict shape.
  - _Fix:_ Move these reads/writes behind typed accessors on Save (Save.seen_mark(code), Save.pops()/Save.bump_pops(), Save.burst_lvl()), so the key strings and the blob shape are owned in one module.
- **[MEDIUM · complexity] _on_giver_tap is a ~80-line method mixing model mutation, reward math, generator grants, and 6 FX/audio side effects** (L1824-1905)
  - Delivering a quest does take/animate the item, remove the quest, credit stars/coins/gems, run a level-up gift + re-sync, place or bag next-map generators, and fire celebrate/floating-reward/audio for each branch, all inline.
